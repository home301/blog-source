## cache stampede
- 부하가 매우 많은 상황에서 캐시가 만료되는 데이터에 대한 요청이 몰리면서 처리 시간 지연 혹은 초과가 발생하는 현상
- 부하가 많은 서비스가 아니라면 성능 테스트에서 특정 리소스에 대한 동시 요청으로 부하를 발생시킬때 캐시 타임아웃이 되는 시점에 발생하는 처리 지연
- 공유 리소스에 특정 시간에 부하를 집중시켜 시스템의 연쇄적인 문제를 발생하게 할 수 있음

### 대응 방법
#### lock
- 실무에서 현실적으로 많이 사용하는 방법
- 캐시 갱신을 위한 lock을 사용, lock을 획득하는 경우에만 캐시 갱신 

#### 외부 재계산
- 캐시 값이 만료에 가까워지면 외부에서 캐시를 갱신하도록 유도
- 관리 포인트가 증가함. 

#### 확률적인 조기 만료
- 캐시가 만료되기 전에 데이터를 갱신하고 만료시간은 연장하는 방법
- 확률 연산을 위한 리소스를 필요
- 확률적이라 정확한 캐시 갱신 시점을 알 수는 없음
- 명시적인 expire를 하는 경우...

### lock 구현 예제 단일
- 2017년에 사용했던 방식 
- 부하가 높은 구간에만 사용(메인 페이지나, 리스트의 첫 10페이지 등)
- 자료구조로 TTL 없는 cached value 와 cache 갱신을 위한 lock 을 TTL을 적용해서 사용
- lock 이 있는 경우 value를 사용 (cached value로 처리)
- lock 이 expire 된 경우 lock을 생성하고 생성에 성공한 경우만 data를 갱신
- lock 획득에 실패한 경우는 lock이 있는 경우와 동일하게 처리 
- 명시적인 expire가 필요하면 lock을 강제로 지우는 방법이 가능(이것도 expire가 매우 빈번하게 발생하는 경우는 별도 대응이 필요한데 그냥 딜레이 협의가 유리)
